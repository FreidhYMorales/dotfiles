## 2.1 Stack de Renderizado

```
┌─────────────────────────────────────────────────────┐
│  HEADER LINE  [yatline]                             │
│  tabs (izq) ────────────────────── hora HH:MM (der) │
├──────────────┬──────────────────────────────────────┤
│              │                                      │
│   PARENT     │   CURRENT DIR    │   PREVIEW PANEL   │
│   (ratio 1)  │   (ratio 3)      │   (ratio 4)       │
│              │                  │                   │
├──────────────┴──────────────────────────────────────┤
│  STATUS LINE  [yatline]                             │
│  mode│size│nombre + count ──── posición│%│ext│perms │
└─────────────────────────────────────────────────────┘
```

**Panel ratio:** `[1, 3, 4]` — el preview ocupa el doble que el directorio actual.

### Por qué yatline (no starship)

| | yatline | starship |
|---|---------|---------|
| Integración | Nativa Lua — acceso a estado interno de Yazi | Shell prompt — sin acceso a estado de Yazi |
| Componentes | `tab_mode`, `hovered_name`, `count`, `permissions`... | Solo variables de shell |
| Modo visual | Cambia color según modo del tab (normal/select/unset) | Sin awareness de modos |
| Dependencias | Nerd Font en Kitty | starship binario externo |
| Transparencia | `show_background = false` → hereda fondo del terminal | Requiere config adicional |

## 2.2 Anatomía de yatline — Componentes

### Header Line

```
LEFT                                              RIGHT
┌─────────────────────────┐          ┌───────────────┐
│  section_a: tabs        │          │  section_a:   │
│  [Tab 1][Tab 2][Tab 3]  │          │  14:32        │
└─────────────────────────┘          └───────────────┘
```

```lua
header_line = {
  left = {
    section_a = {
      { type = "line", custom = false, name = "tabs", params = { "left" } },
    },
  },
  right = {
    section_a = {
      { type = "string", custom = false, name = "date", params = { "%H:%M" } },
    },
  },
}
```

### Status Line

```
LEFT                                                      RIGHT
┌────────┬──────────┬──────────────────────────┐  ┌──────┬────┬────────────┐
│ [mode] │ [size]   │ filename.ext  [3 sel]     │  │ 5/42 │ 11%│ .lua [rwx] │
└────────┴──────────┴──────────────────────────┘  └──────┴────┴────────────┘
section_a  section_b  section_c                    section_a  section_b  section_c
```

| Componente | Tipo | Dato que muestra |
|------------|------|-----------------|
| `tab_mode` | string | NORMAL / SELECT / UN_SET |
| `hovered_size` | string | Tamaño del archivo bajo cursor |
| `hovered_name` | string | Nombre del archivo |
| `count` | coloreds | Archivos seleccionados / copiados / cortados |
| `cursor_position` | string | Posición actual / total (ej: 5/42) |
| `cursor_percentage` | string | Porcentaje de scroll |
| `hovered_file_extension` | string | Extensión del archivo |
| `permissions` | coloreds | Permisos rwx con colores |

## 2.3 Sistema de Colores

### Arquitectura actual: Flavor "deadlock"

```toml
# theme.toml
[flavor]
dark  = "deadlock"
light = "deadlock"
```

Los colores del file manager, highlights, selección, etc. están definidos en el flavor file "deadlock". Para modificar colores, editar el flavor, no `theme.toml`.

### Convención crítica: Lua sigue su propia API (Invariante I-02)

Los colores en `init.lua` (yatline, etc.) usan la API Lua de Yazi — convención distinta al TOML:

| Contexto | Formato correcto | Formato incorrecto |
|----------|-----------------|-------------------|
| `theme.toml` / flavor | `"bright-black"` (con guion) | `"brightblack"` |
| `init.lua` (Lua) | `"brightblack"` (sin guion) | `"bright-black"` |

El error no produce warning — el color simplemente no aplica.

### Paleta ANSI en init.lua (yatline)

```
Estándar:   black, red, green, yellow, blue, magenta, cyan, white
Bright:     brightblack, brightred, brightgreen, brightyellow,
            brightblue, brightmagenta, brightcyan, brightwhite
```

### Nota histórica

La config comenzó con ANSI puro (`use = ""`). Se migró a flavor "deadlock" para resolver el problema del highlight de selección con esquinas redondeadas. El flavor permite control más preciso sobre el renderizado del cursor/hover row.

### Esquema de colores actual en yatline

```lua
style_a = {
  fg = "black",
  bg_mode = {
    normal = "blue",    -- modo normal → fondo azul
    select = "green",   -- modo selección → fondo verde
    un_set = "red",     -- modo sin set → fondo rojo
  },
},
style_b = { bg = "brightblack", fg = "white"      },  -- sección secundaria
style_c = { bg = "black",       fg = "brightwhite" },  -- texto / fondo del bar
```

### Colores de permisos

```lua
permissions_t_fg = "green",   -- tipo (d, l, -)
permissions_r_fg = "yellow",  -- lectura (r)
permissions_w_fg = "red",     -- escritura (w)
permissions_x_fg = "cyan",    -- ejecución (x)
permissions_s_fg = "white",   -- setuid/setgid (s)
```

## 2.4 Separadores Powerline

```lua
section_separator = { open = "", close = "" },  -- entre secciones
part_separator    = { open = "", close = "" },  -- entre partes
inverse_separator = { open = "", close = "" },  -- separador inverso
```

Requieren **Nerd Font** activa en Kitty. Sin Nerd Font, aparecen como caracteres corruptos — no como espacios en blanco.

**Verificar en Kitty:** `~/.config/kitty/kitty.conf` → `font_family` debe ser una Nerd Font (ej: `JetBrainsMono Nerd Font`).

## 2.5 Orden de carga en init.lua (Invariante I-03)

El orden importa. Cargar un plugin antes de que sus dependencias estén listas puede causar crashes silenciosos.

```
01. yatline          → require():setup()  — UI primero, siempre
02. clipboard        → sin require()      — keybind-only
03. relative-motions → require():setup()
04. bypass           → sin require()      — keybind-only
05. fg               → require():setup()
06. gvfs             → require():setup()
07. mount            → sin require()      — keybind-only
[piper]              → sin require()      — previewer puro
08. rich-preview     → sin require()      — previewer puro
09. glow             → sin require()      — via piper
10. hexyl            → sin require()      — via piper
11. mediainfo        → sin require()      — previewer puro
12. ouch             → sin require()      — previewer + keybind
13. what-size        → require():setup()
14. lazygit          → sin require()      — keybind-only
15. recycle-bin      → require():setup()
```

### Regla de detección — ¿necesita setup()?

```bash
# Leer la última línea de main.lua del plugin
tail -1 ~/.config/yazi/plugins/<plugin>.yazi/main.lua

# return { entry = ... }   → tabla plana → solo require("plugin")
# return M                 → módulo con M:setup() → require("plugin"):setup({})
```

**Nunca** llamar `:setup()` en plugins que retornan tabla plana — produce runtime crash.
